# smart-calendar

AI-ассистент для управления Google Calendar через естественный язык. Пользователь общается с приложением через браузерный чат; бэкенд запускает агентный цикл, который преобразует запросы в вызовы Google Calendar API.

## Что делает приложение

- Аутентифицирует пользователя в Google через OAuth2 и сохраняет токен в HTTP-only cookie.
- Принимает сообщения на естественном языке через WebSocket-соединение.
- Запускает агентный цикл на базе LLM (Claude claude-opus-4-7 с адаптивным мышлением), который решает, что делать: вызвать инструмент календаря, задать уточняющий вопрос или дать финальный ответ.
- Стримит ответ LLM токен за токеном в браузер, чтобы пользователь видел текст в реальном времени.
- Раздаёт одностраничный чат-интерфейс, встроенный прямо в бинарник через `go:embed`.

## Архитектура

Проект следует принципам чистой архитектуры: каждый слой зависит только от абстракций, но не от конкретных реализаций других слоёв.

```
cli/
  root.go                     -- точка входа Cobra; загружает конфиг, собирает приложение, обрабатывает сигналы

internal/
  application/
    app.go                    -- сборка верхнего уровня: создаёт Server, устанавливает RedirectURL

  config/
    config.go                 -- загрузка YAML-конфига

  models/
    agent.go                  -- типы агента: Message, ToolCall, AgentAction, Chunk, StreamEvent
    calendar.go               -- доменные типы: Event, Calendar, TimeSlot, SuggestTimeRequest

  service/
    interfaces.go             -- интерфейсы CalendarClient и LLMClient
    smart_calendar_service.go -- агентный цикл: Execute, runTurn, readStream, dispatchTool

  adapters/clients/
    claude/
      config.go               -- конфиг Claude (APIKey, BaseURL)
      client.go               -- реализация LLMClient; стриминг через Anthropic SDK
      tool.go                 -- определения схем инструментов, передаваемых Claude

    google/
      config.go               -- конфиг OAuth2 для Google
      auth.go                 -- вспомогательные функции AuthURL и ExchangeCode
      client.go               -- реализация CalendarClient поверх Google Calendar API v3

  controller/restapi/v1/
    server.go                 -- настройка HTTP-сервера, регистрация маршрутов
    auth.go                   -- обработчики /auth/google и /oauth2callback
    ws.go                     -- WebSocket-обработчик; создаёт сервис для каждого соединения
    ui.go                     -- раздаёт встроенный index.html; перенаправляет неавторизованных
    helpers.go                -- работа с cookie, кодировщик chunkFrame, управление state map
    static/index.html         -- чат-интерфейс (go:embed)
```

### Направление зависимостей

```
controller  -->  service interfaces  -->  adapters
                       |
                    models
```

Пакет `service` определяет интерфейсы `CalendarClient` и `LLMClient`. Адаптеры (`google`, `claude`) реализуют эти интерфейсы. Контроллер знает только о сервисе — не об Anthropic SDK и не о Google SDK. Это значит, что любой адаптер можно заменить или подменить заглушкой, не трогая бизнес-логику.

## Агентный цикл

`SmartCalendarService.Execute` — точка входа в сессию разговора. Вызывается один раз на каждое WebSocket-соединение и живёт всё время его существования.

```
inputCh  <-chan string      -- сюда приходят сообщения от пользователя
outputCh  chan models.Chunk -- отсюда уходят стриминговые фрагменты ответа
```

### Execute

`Execute` запускает горутину, которая ведёт историю сообщений и в цикле обрабатывает входящие сообщения пользователя:

История — это плоский срез `[]models.Message` с ролями: `system`, `user`, `assistant`, `tool`. Она накапливается на протяжении всей сессии, чтобы модель всегда имела полный контекст разговора.

### runTurn

`runTurn` управляет LLM до 20 шагов на одно сообщение пользователя. Каждый шаг:

1. Вызывает `s.llm.NextAction(ctx, history)` — возвращает `<-chan models.StreamEvent`.
2. Вызывает `readStream`, который вычитывает канал:
   - `StreamEvent{Chunk: "..."}` — пересылается в `outputCh` как `Chunk{Text: "..."}`
   - `StreamEvent{Action: &action}` — стрим завершён, возвращается действие
   - `StreamEvent{Err: err}` — прерывается с ошибкой
3. Делает switch по возвращённому действию:

| Тип действия         | Что происходит                                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `FinalAnswer != ""`  | Добавляется в историю как `assistant`; в output отправляется `Chunk{Done: true}`; ход завершается                         |
| `ClarifyingQuestion` | Отправляется пользователю; ход блокируется на `inputCh` в ожидании ответа; ответ добавляется в историю; цикл продолжается |
| `ToolCalls`          | Каждый инструмент исполняется; результаты добавляются в историю с `role=tool`; цикл переходит к следующему шагу           |

### Вызов инструментов

`dispatchTool` — простой switch по `call.Name`. Он десериализует JSON-аргументы, которые сгенерировал Claude, вызывает соответствующий метод `CalendarClient` и сериализует результат обратно в JSON. Этот JSON сохраняется в историю как сообщение `tool` и на следующем шаге передаётся Claude.

Доступные инструменты:

| Инструмент          | Операция с календарём              |
| ------------------- | ---------------------------------- |
| `create_event`      | Создать событие                    |
| `delete_event`      | Удалить по ID                      |
| `get_event`         | Получить одно событие              |
| `list_calendars`    | Перечислить календари              |
| `list_events`       | Список событий в диапазоне времени |
| `respond_to_event`  | Принять / отклонить / под вопросом |
| `suggest_time`      | Найти свободные слоты для группы   |
| `update_event`      | Изменить существующее событие      |
| `ask_clarification` | Задать вопрос пользователю         |

`ask_clarification` — особый инструмент: он определён как tool, чтобы Claude мог явно вызвать его, когда не хватает информации. Сервис перехватывает его до передачи в calendar client, отправляет вопрос пользователю, блокируется на `inputCh` и продолжает цикл с ответом, добавленным в историю.

## Каналы

Каждый этап пайплайна общается через каналы; в каждой блокирующей точке присутствует отмена через контекст.

### Стриминг LLM: `<-chan models.StreamEvent`

`LLMClient.NextAction` сразу возвращает канал. Горутина внутри Claude-адаптера:

1. Открывает SSE-стрим к Anthropic API.
2. Итерирует события `stream.Next()`, накапливает блоки контента по индексу.
3. Для каждого `TextDelta` на блоке типа `text` отправляет `StreamEvent{Chunk: delta}`.
4. Когда стрим заканчивается, собирает `AgentAction` из накопленных блоков и отправляет `StreamEvent{Action: &action}`.
5. При любой ошибке отправляет `StreamEvent{Err: err}`.
6. Закрывает канал.

Канал буферизован (ёмкость 32), чтобы горутина не блокировалась, если сервис временно занят пересылкой фрагментов. Блоки `thinking` из адаптивного мышления накапливаются, но не пересылаются — пользователь видит только финальный текст.

### Выход сервиса: `chan models.Chunk`

`Execute` возвращает `chan models.Chunk`. По этому каналу идут два вида значений:

- `Chunk{Text: "..."}` — по одному на каждый текстовый фрагмент, напрямую из `StreamEvent`.
- `Chunk{Done: true}` — отправляется после завершения каждого хода (финальный ответ или уточняющий вопрос), сигнализирует браузеру разблокировать поле ввода.

### WebSocket-слой

Обработчик WS запускает отдельную горутину записи, которая перебирает `outputCh`:

```go
for chunk := range outputCh {
    conn.WriteMessage(websocket.TextMessage, chunkFrame(chunk))
}
```

`chunkFrame` кодирует фрагмент в компактный JSON-фрейм:

- `{"t":"c","v":"текстовый фрагмент"}` — для текста
- `{"t":"e"}` — конец ответа

Главная горутина обработчика WS только читает из WebSocket и пересылает сообщения в `inputCh`. В каждый момент есть ровно один читатель и один писатель на WebSocket-соединение — это удовлетворяет требованиям конкурентности gorilla/websocket.

### Отмена через контекст

Каждый блокирующий `select` в сервисе содержит `case <-ctx.Done()`. Когда WebSocket-соединение закрывается, обработчик вызывает `cancel()`, что распространяется через контекст в LLM-стрим (из-за чего `stream.Next()` возвращает false) и во все вызовы `sendChunk` / `recv` в горутине сервиса.

## Поток авторизации Google OAuth2

1. `GET /` — сервер проверяет cookie `google_token`. Если отсутствует, перенаправляет на `/auth/google`.
2. `GET /auth/google` — генерирует случайный 16-байтный nonce состояния, сохраняет его в in-memory map с временем жизни 10 минут, перенаправляет на страницу согласия Google.
3. `GET /oauth2callback` — проверяет nonce (читается и удаляется из map), обменивает code на токен, сохраняет токен как hex-encoded JSON в HTTP-only cookie (`SameSite=Lax`, max-age 15 минут), перенаправляет на `/`.
4. Последующие запросы к `/` отдают чат-интерфейс. Обработчик WS читает токен из cookie и создаёт клиент Google Calendar для каждого соединения.

## Конфигурация

```yaml
http:
  port: 8080

google:
  client_id: "your-google-client-id.apps.googleusercontent.com"
  client_secret: "your-google-client-secret"

claude:
  api_key: "sk-ant-..."
  base_url: "https://api.anthropic.com"
```

`RedirectURL` для Google OAuth2 формируется автоматически из настроенного порта и не требует явного указания в конфиге.

### Флаги CLI

```
-c, --config  путь к YAML-файлу конфигурации (по умолчанию: config.yaml)
-p, --port    порт HTTP-сервера, переопределяет значение из конфига
```

## Запуск

```bash
go run . -c config.yaml
```

Открыть `http://localhost:8080` в браузере. При первом посещении приложение перенаправит на страницу согласия Google OAuth. После авторизации откроется чат-интерфейс.
